# AGENTS.md — CQU Compiler Principles Course Project

This repository contains a three-stage compiler construction project (bex1, bex2, bex3) for Chongqing University's Compiler Principles course. The source language is **SysY** (a C-like educational language, files ending in `.sy`). The target is **RISC-V RV32I** assembly.

## Project Structure

Each `bexN/` directory is an independent CMake project with the same layout:

```
bexN/
  CMakeLists.txt       # Build config (C++14)
  main.cpp             # CLI entry point, dispatches by -step flag
  include/             # Public headers
    front/             # Lexical, syntax, semantic headers
    ir/                # IR data structures (Operand, Instruction, Function, Program)
    backend/           # Code generator + RISC-V definitions (bex3 only)
    tools/             # IR executor / reader (bex2/bex3 only)
    json/              # jsoncpp headers
  src/
    front/             # lexical.cpp, syntax.cpp, semantic.cpp, token.cpp, ast.cpp
    ir/                # IR implementation (bex2/bex3; or linked from lib in bex1)
    backend/           # generator.cpp, rv_def.cpp, rv_inst_impl.cpp (bex3 only)
    tools/             # ir_executor.cpp (bex2/bex3)
    third_party/jsoncpp/
  lib/                 # Prebuilt static libraries
    libIR-x86-win.a
    libIR-x86-linux.a
    libTools-x86-win.a  (bex2/bex3)
    libTools-x86-linux.a (bex2/bex3)
  test/                # Python test harness
    build.py
    run.py
    score.py
    test.py
    testcase/          # .sy source files + .in inputs + .out expected outputs
      basic/
      function/
    ref/               # Reference outputs per stage (s0, s1, s2, S)
    output/            # Generated by run.py (gitignored)
```

### Stage Responsibilities

| Stage | What it does | Key new code |
|-------|-------------|--------------|
| **bex1** | Lexical analysis + Syntax analysis | `frontend::Scanner` (DFA), `frontend::Parser` (recursive descent), AST → JSON |
| **bex2** | + Semantic analysis + IR generation | `frontend::Analyzer`, symbol table, `ir::Program`, IR executor (`-e`) |
| **bex3** | + RISC-V backend code generation | `backend::Generator`, register allocation, stack frame, RV32I + float |

Stages build on each other: copy `src/front/` and `include/front/` forward when advancing.

## Build System

- **Toolchain**: CMake 3.10+, C++14.
- **Windows**: Uses `MinGW Makefiles` generator (`mingw32-make`). The `test/build.py` automatically passes `-G "MinGW Makefiles"` on Windows.
- **Linux**: Standard `cmake .. && make`.
- **Output**: Executable `compiler` goes to `bexN/bin/`. Object archives go to `bexN/bin/` too.

### Build commands

```bash
cd bexN/build
cmake ..          # On Windows: cmake -G "MinGW Makefiles" ..
make              # On Windows: mingw32-make
```

Or use the Python helper from inside `test/`:

```bash
cd bexN/test
python build.py
```

**Critical**: CMake in bex1/bex3 links prebuilt static libraries from `lib/`. On the OJ/grading system, `libIR.a` and `libTools.a` are expected. For local builds:
- bex2/bex3 CMakeLists.txt already auto-selects `libIR-x86-win.a` / `libTools-x86-win.a` on Windows and the `-linux` variants elsewhere.
- bex1's CMakeLists.txt expects you to manually rename the platform-specific `.a` to `libIR.a` if you uncomment the generic link line.

## Running the Compiler

```bash
./compiler <src.sy> -<step> -o <output> [-O1]
```

Steps:
- `-s0`: Token stream (one per line: `TokenType\tvalue`)
- `-s1`: AST as JSON (`Json::StyledWriter`)
- `-s2`: IR text (`ir::Program::draw()`)
- `-e`: Execute the IR interpreter and append the main return value to output (bex2/bex3)
- `-S`: RISC-V assembly (bex3)

## Testing

All testing is driven by Python scripts in each `test/` directory.

```bash
cd bexN/test
python test.py <s0|s1|s2|S>    # build + run + score in one shot
python build.py                # cmake + make
python run.py <s0|s1|s2|S>     # compile all testcases, write to output/
python score.py <s0|s1|s2|S>   # diff against ref/
```

### Test case layout

Each `.sy` file may have a matching `.in` (stdin) and `.out` (expected stdout + return code on last line). Test cases live in:
- `testcase/basic/` — simpler constructs
- `testcase/function/` — advanced features (arrays, functions, short-circuit, I/O)

### Scoring quirks

- `score.py` hardcodes `total = 47` (bex1) or `total = 58` (bex3). If you add/remove test cases, update that constant.
- For `-s0`/`-s1` in bex1, scoring normalizes whitespace before comparing. For `-s1`, it also tries strict JSON equality first.
- For bex3 `-S` scoring: `score.py` compiles the generated `.s` with `riscv32-unknown-linux-gnu-gcc`, links against `sylib-riscv-linux.a`, runs under `qemu-riscv32.sh`, and diffs the resulting `.out` against `ref/s2/` (not `ref/S/`).
- `qemu-riscv32.sh` is **not** in this repo; it is provided by the course environment.

## Code Patterns & Conventions

### Namespaces
- `frontend::` — scanner, parser, AST nodes, semantic analyzer
- `ir::` — intermediate representation
- `backend::` — code generator (bex3)
- `rv::` — RISC-V register/instruction enums (bex3)

### AST Design
- Base class: `frontend::AstNode` (has `type`, `parent`, `children`).
- Every grammar non-terminal has a subclass (e.g., `CompUnit`, `FuncDef`, `Stmt`, `Exp`).
- AST nodes are **not** copyable; constructors/delete are disabled.
- Many expression nodes carry:
  - `bool is_computable` — whether the subexpression is compile-time constant
  - `string v` — the constant value (if computable)
  - `Type t` — the evaluated type

### IR Design
- `ir::Operand` = `{ name, type }` where `type` is one of `Int`, `Float`, `IntLiteral`, `FloatLiteral`, `IntPtr`, `FloatPtr`, `null`.
- `ir::Instruction` = `{ op1, op2, des, op }`.
- `ir::CallInst` extends `Instruction` with `argumentList`.
- `ir::Function` = name, return type, parameter list, instruction vector.
- `ir::Program` = list of functions + global values.
- IR string format is consumed by the grading executor; do not change `draw()` output semantics.

### Semantic Analyzer Patterns
- `frontend::Analyzer` is a singleton-like struct with `tmp_cnt` for temporary variable naming and `g_init_inst` for global initializers.
- `SymbolTable` maintains a `scope_stack` of `ScopeInfo` blocks. Use `get_scoped_name()` to disambiguate shadowed identifiers.
- A common macro in semantic.cpp: `#define GET_CHILD_PTR(node, type, index) auto node = dynamic_cast<type*>(root->children[index]); assert(node);`
- Scope names often follow the pattern `{name}_block` for nested blocks.
- Library functions (`getint`, `putint`, `getfloat`, `putfloat`, `getch`, `putch`, `getarray`, `putarray`, `putf`, `starttime`, `stoptime`) are pre-seeded into the symbol table via `get_lib_funcs()`.

### Backend Patterns (bex3)
- `backend::Generator` walks `ir::Program` function-by-function, instruction-by-instruction.
- **Stack layout**: `s0` (frame pointer) is used. Each function prologue saves `ra` and `s0`, sets up `s0 = sp + frame_size`, and epilogue restores them.
- **Frame size** is 16-byte aligned.
- **Global variables**: prefixed with `_g_` in assembly (e.g., `_g_x`). Stored in `.data` section.
- **Register usage**: `t0`–`t2` / `ft0`–`ft2` are used as temporaries. Arguments use `a0`–`a7` / `fa0`–`fa7`.
- **Calling convention**: Arguments beyond the 8th are spilled to the stack. The generator assigns argument locations with `assign_arg_locs()`.
- **Load/store helpers**: `load_int`, `store_int`, `load_float`, `store_float`, `load_ptr`, `store_ptr` handle globals vs locals automatically.

## Important Gotchas

1. **Lib linking for grading**: The OJ links against `libIR.a` and `libTools.a`. For local debug, bex2/bex3 CMakeLists have commented-out sections to build IR/Tools from source instead of linking the prebuilt `.a`. **Do not** ship modified IR executor source; the grader uses the prebuilt library.

2. **Windows path handling**: `test/run.py` and `test/score.py` replace `/` with `\\` on Windows when shelling out.

3. **Return code handling**: For `-e` and for `-S` under QEMU, the main return value is appended as a trailing `\n<returncode>` to the output file. The test harness expects this.

4. **Float literals in IR/backend**: Floats are carried as strings. In the backend, `float_bits()` uses `memcpy` to bit-cast float → `uint32_t`, then `li` + `fmv.w.x` to load into float registers.

5. **Integer literal parsing**: The compiler accepts decimal, hex (`0x`), octal (`0` prefix), and binary (`0b`). Both frontend and backend duplicate parsing logic for literals.

6. **Short-circuit evaluation**: Test cases like `50_short_circuit.sy` validate that `&&` and `||` do not evaluate the right-hand side when unnecessary. The semantic analyzer / IR generator must emit conditional jumps correctly.

7. **Break/continue**: `frontend::Stmt` has `jump_eow` and `jump_bow` sets of `ir::Instruction*` pointers. These are backpatched after the while loop body is generated.

8. **JSON output**: AST JSON uses `Json::StyledWriter` (not `FastWriter`), which produces pretty-printed output with trailing newlines. The `-s1` grader normalizes this, but local diff may flag whitespace.

9. **No git in bex2/bex3**: Only `bex1/` is a git repository. If you need version control for later stages, initialize it manually.

10. **RISC-V toolchain**: The `riscv-gnu-toolchain/` inside `bex3/test/` is a full submodule tree but is **not** needed for building the compiler itself. It is only there for reference or if you want to build the cross-toolchain locally.

## Recommended Workflow for Agents

1. Identify which stage (`bex1/2/3`) the user is working on.
2. Build with `python test/build.py` from the stage's `test/` directory.
3. Run a single step: `python test/run.py s0` (or `s1`, `s2`, `S`).
4. Score: `python test/score.py s0`.
5. For debugging a single test, run the compiler directly:
   ```bash
   cd bex3/bin
   ./compiler ../test/testcase/basic/00_main.sy -S -o /tmp/00_main.s
   ```
6. When modifying frontend code, remember to propagate changes forward across stages (bex1 → bex2 → bex3) if the user intends to keep earlier stages consistent.

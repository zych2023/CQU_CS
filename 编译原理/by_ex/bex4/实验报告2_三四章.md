## 三、实验过程或算法（源程序）

### 3.1 实验 0：环境与基线（O0/O2 + 生成汇编验证优化发生）

#### 3.1.1 实验目的

本实验旨在验证毕昇编译器在不同优化等级（`-O0` 与 `-O2`）下对同一段 C 代码的编译行为差异。通过对比运行输出、汇编代码和基本计时，建立后续实验的评估基线，并直观观察常量折叠（Constant Folding）、死分支消除（Dead Branch Elimination）等基础优化的实际效果。

#### 3.1.2 源程序说明

实验源程序为 `toy_opt.c`，其核心逻辑非常简单，便于观察编译器优化痕迹：

```c
#include <stdio.h>
__attribute__((noinline)) int bar() {
    int a = 10 * 10;
    int b = 20;
    if (a > 0) return a + 1;
    else return a - 1;
}
int main() {
    volatile int x = bar();
    printf("%d\n", x);
    return 0;
}
```

程序中 `bar()` 函数内部包含一个编译期即可确定的乘法 `10*10`，以及一个恒为真的条件分支 `if (a > 0)`。`__attribute__((noinline))` 强制编译器不内联 `bar()`，以便在汇编中仍能定位到该函数的代码边界；`volatile int x` 则阻止 `main()` 将 `bar()` 的结果完全优化掉，保证 `printf` 一定会被执行。

#### 3.1.3 编译与运行步骤

在 Docker 容器内的 `exp0_baseline` 目录下执行以下命令：

```bash
# 1. 分别在 -O0 和 -O2 下编译并运行
clang -O0 toy_opt.c -o toy_O0 && ./toy_O0
clang -O2 toy_opt.c -o toy_O2 && ./toy_O2

# 2. 生成对应汇编文件
clang -O0 -S toy_opt.c -o toy_O0.s
clang -O2 -S toy_opt.c -o toy_O2.s

# 3. 对比汇编差异（重点观察 bar() 函数）
diff -u toy_O0.s toy_O2.s

# 4. 多次计时，取中位数
for i in 1 2 3; do time -p ./toy_O0; done
for i in 1 2 3; do time -p ./toy_O2; done
```

### 3.2 实验 1：自动并行化（SIMD/NEON 向量化 + OpenMP 并行 for）

#### 3.2.1 实验目的

验证毕昇编译器在 AArch64 平台上对浮点数组运算的自动向量化能力，并对比三种实现方式：
1. 显式禁用向量化（`#pragma clang loop vectorize(disable)`）的标量版本；
2. 由编译器自动向量化的 SIMD/NEON 版本；
3. 使用 OpenMP 多线程并行的版本。

通过计时对比和汇编指令分析，区分“指令级并行（SIMD）”与“线程级并行（OpenMP）”在性能上的不同贡献。

#### 3.2.2 源程序说明

实验包含三个源文件，核心计算均为向量加法 `out[i] = a[i] + b[i]`，仅在并行策略上有所区别。

**（1）`vec_add.c` —— 禁止向量化（no-vec）**

```c
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
void __attribute__ ((noinline))
f(int N, float *a, float *b, float *out) {
#pragma clang loop vectorize(disable)
    for (int i = 0; i < N; i++) out[i] = a[i] + b[i];
}
int main() {
    int loop = 1000;
    int len  = 200000;
    float *a = (float*)malloc(sizeof(float) * len);
    float *b = (float*)malloc(sizeof(float) * len);
    float *o = (float*)malloc(sizeof(float) * len);
    for (int i = 0; i < len; i++) { 
        a[i] = (float)rand() /((float)RAND_MAX + 1.0f);
        b[i] = (float)rand() / ((float)RAND_MAX + 1.0f); 
    }
    clock_t st = clock();
    for (int i = 0; i < loop; i++) f(len, a, b, o);
    clock_t ed = clock();
    double t = (double)(ed - st) / CLOCKS_PER_SEC;
    printf("time: %f s\n", t);
    volatile float guard = o[0];
    (void)guard;
    free(a); free(b); free(o);
    return 0;
}
```

通过 `#pragma clang loop vectorize(disable)` 强制编译器生成标量循环，作为性能对比的基线。

**（2）`vec_add_vec.c` —— 自动向量化（vec）**

与 `vec_add.c` 的区别仅在于去掉了 `vectorize(disable)` 的 pragma，允许编译器在 `-O2` 下自动尝试 NEON 向量化。

**（3）`vec_add_omp.c` —— OpenMP 并行（omp）**

在 `vec_add_vec.c` 的基础上，在循环前增加 `#pragma omp parallel for`，并使用 `-fopenmp` 编译，以启动多线程并行。

#### 3.2.3 编译与运行步骤

```bash
cd ~/workspace/bisheng_lab/exp1_parallel

# 编译三个版本
clang -O2 vec_add.c -o vec_no_vec
clang -O2 vec_add_vec.c -o vec_vec
clang -O2 -fopenmp vec_add_omp.c -o vec_omp

# 运行计时（各 3 次）
for i in 1 2 3; do time -p ./vec_no_vec; done
for i in 1 2 3; do time -p ./vec_vec; done
for i in 1 2 3; do time -p ./vec_omp; done

# 生成汇编以提取向量化证据
clang -O2 -S vec_add.c -o vec_no_vec.s
clang -O2 -S vec_add_vec.c -o vec_vec.s
```

### 3.3 实验 2：内存布局优化（AoS vs SoA）

#### 3.3.1 实验目的

验证数据布局（Data Layout）对编译器向量化和内存访存效率的影响。重点对比：
1. **AoS（Array of Structs）**：结构体数组，访问单个字段时跨步大；
2. **SoA（Struct of Arrays）**：将同名字段拆分为独立数组，实现连续访存。

同时简要观察 `struct-peel` 优化的代码生成变化。

#### 3.3.2 源程序说明

**（1）`aos_soa.c` —— AoS 与 SoA 对比**

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <time.h>
#define N 1200000
#define REPEAT 10
typedef struct {
    float a;
    float pad[31];
} S;
static inline double now_sec() {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return (double)ts.tv_sec + (double)ts.tv_nsec * 1e-9;
}
__attribute__((noinline))
void transform_a_aos(const S* x, float* out) {
    for (int r = 0; r < REPEAT; r++) {
        for (int i = 0; i < N; i++) {
            out[i] = x[i].a * 1.1f + 2.0f;
        }
    }
}
__attribute__((noinline))
void transform_a_soa(const float* a, float* out) {
    for (int r = 0; r < REPEAT; r++) {
        for (int i = 0; i < N; i++) {
            out[i] = a[i] * 1.1f + 2.0f;
        }
    }
}
int main() {
    S* xs = (S*)aligned_alloc(64, sizeof(S) * (size_t)N);
    float* a = (float*)aligned_alloc(64, sizeof(float) * (size_t)N);
    float* out_aos = (float*)aligned_alloc(64, sizeof(float) * (size_t)N);
    float* out_soa = (float*)aligned_alloc(64, sizeof(float) * (size_t)N);
    for (int i = 0; i < N; i++) {
        xs[i].a = (float)i * 0.001f;
        memset(xs[i].pad, 0, sizeof(xs[i].pad));
        a[i] = xs[i].a;
        out_aos[i] = 0.0f;
        out_soa[i] = 0.0f;
    }
    double st1 = now_sec();
    transform_a_aos(xs, out_aos);
    double ed1 = now_sec();
    double st2 = now_sec();
    transform_a_soa(a, out_soa);
    double ed2 = now_sec();
    double checksum_aos = 0.0, checksum_soa = 0.0;
    for (int i = 0; i < N; i++) {
        checksum_aos += out_aos[i];
        checksum_soa += out_soa[i];
    }
    printf("checksum_aos=%0.3f checksum_soa=%0.3f\n", checksum_aos, checksum_soa);
    printf("time_aos=%f time_soa=%f\n", ed1 - st1, ed2 - st2);
    free(xs); free(a); free(out_aos); free(out_soa);
    return 0;
}
```

`S` 结构体包含 1 个有效字段 `a` 和 31 个填充字段，总大小为 128B。在 AoS 布局下，访问 `x[i].a` 的内存步长即为 128B；而在 SoA 布局下，`a[]` 是纯 `float` 数组，访存完全连续。

**（2）`struct_peel.c` —— struct-peel 验证（简要）**

该文件用于观察 `-O3 -flto` 下编译器是否能通过结构体剥离优化减少动态内存分配次数。编译命令如下：

```bash

```

### 3.4 实验 3：PGO 反馈优化

#### 3.4.1 实验目的

验证基于配置文件引导的优化（Profile-Guided Optimization, PGO）在实际排序程序中的效果。PGO 的核心流程为：插桩编译 → 运行采集分支/调用频率 → 合并 profile → 反馈编译。通过与基线 `-O2` 对比，观察 PGO 对代码布局和分支预测的影响。

#### 3.4.2 源程序说明

实验源程序为 `bubble_sort.c`，采用冒泡排序对升序数组进行处理：

```c
#include <stdio.h>
#include <sys/time.h>
#define ARRAY_LEN 80000
int data[ARRAY_LEN];
static struct timeval tm1;
static inline void start() { gettimeofday(&tm1, NULL); }
static inline void stop() {
  struct timeval tm2; gettimeofday(&tm2, NULL);
  unsigned long long t = 1000 * (tm2.tv_sec - tm1.tv_sec) + 
  (tm2.tv_usec - tm1.tv_usec) / 1000;
  printf("Average: %llu ms\n", t);
}
void swap_int(int *a,int *b){ int c=*a; *a=*b; *b=c; }
void bubble_sort(int *a, int n){
  for(int i=0;i<n-1;i++){
    for(int j=i+1;j<n-1;j++)
      if(a[i] < a[j]) swap_int(&a[i],&a[j]);
  }
}
void get_data(){
  for(int i=0;i<ARRAY_LEN;i++) data[i]=i;
#ifdef __aarch64__
  asm volatile("DMB SY");
#endif
}
int main(){
  get_data();
  start();
  bubble_sort(data, ARRAY_LEN);
  stop();
  return 0;
}
```

`get_data()` 初始化升序数组 `data[i]=i`，并插入 `DMB SY` 内存屏障确保数据可见性。由于输入为升序，冒泡排序的内层循环中 `a[i] < a[j]` 分支具有高度可预测性，这正是 PGO 能够利用的 profile 特征。

#### 3.4.3 编译与运行步骤

```bash
cd ~/workspace/bisheng_lab/exp3_pgo

# 1. 基线编译运行
clang -O2 bubble_sort.c -o base.exe
for i in 1 2 3; do time -p ./base.exe; done

# 2. 插桩编译 + 收集 profile
clang -O2 -fprofile-instr-generate bubble_sort.c -o sample.exe
LLVM_PROFILE_FILE="code-%p.profraw" ./sample.exe >/dev/null

# 3. 合并 profile
llvm-profdata merge -output=code.profdata code-*.profraw

# 4. 反馈编译运行
clang -O2 -fprofile-instr-use=code.profdata bubble_sort.c -o pgo.exe
for i in 1 2 3; do time -p ./pgo.exe; done

# 5. 反汇编对比
objdump -d base.exe > base.dump
objdump -d pgo.exe > pgo.dump
```

### 3.6 实验 5：综合循环体优化（Loop Fusion + Autotuner 对比）

#### 3.6.1 实验目的

验证循环融合（Loop Fusion）对减少内存访问开销和提升 Cache 利用率的作用，并对比不同优化等级（`-O0`、`-O1`、`-O2`）以及 Autotuner 自动调优的效果。通过汇编中 `Inner Loop Header` 的数量变化，量化循环融合对循环结构的影响。

#### 3.6.2 源程序说明

实验源程序为 `loop_fusion_test.c`，通过宏 `FUSED` 控制是否启用循环融合：

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#define N 6000000
#define REPEAT 3
static inline double now_sec() {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return (double)ts.tv_sec + (double)ts.tv_nsec * 1e-9;
}
int main() {
    float *a = (float*)aligned_alloc(64, (size_t)N * sizeof(float));
    float *b = (float*)aligned_alloc(64, (size_t)N * sizeof(float));
    float *c = (float*)aligned_alloc(64, (size_t)N * sizeof(float));
    float *d = (float*)aligned_alloc(64, (size_t)N * sizeof(float));
    for (int i = 0; i < N; i++) { a[i] = (float)i; b[i] = (float)(N - i); }
    double st = now_sec();
#if defined(FUSED)
    for (int r = 0; r < REPEAT; r++) {
        for (int i = 0; i < N; i++) {
            float ai = a[i];
            c[i] = ai + b[i];
            d[i] = ai * 0.5f;
        }
    }
#else
    for (int r = 0; r < REPEAT; r++) {
        for (int i = 0; i < N; i++) c[i] = a[i] + b[i];
        for (int i = 0; i < N; i++) d[i] = a[i] * 0.5f;
    }
#endif
    double ed = now_sec();
    float checksum = 0.0f;
    for (int i = 0; i < N; i++) checksum += c[i] + d[i];
    fprintf(stderr, "Checksum: %f\n", checksum);
#if defined(FUSED)
    printf("mode=fused ");
#else
    printf("mode=unfused ");
#endif
    printf("N=%d REPEAT=%d time=%f\n", N, REPEAT, (ed - st));
    free(a); free(b); free(c); free(d);
    return 0;
}
```

- **unfused 版本**：每次外循环包含两个独立的内循环，分别计算 `c[i]` 和 `d[i]`，数组 `a[]` 被遍历两次；
- **fused 版本**：将两次遍历合并为一个内循环，同时计算 `c[i]` 和 `d[i]`，`a[]` 仅遍历一次。

#### 3.6.3 编译与运行步骤

```bash

# 编译各版本
clang -O0 loop_fusion_test.c -o unfused_O0

# 运行各 3 次
./unfused_O0 && ./unfused_O0 && ./unfused_O0
./unfused_O1 && ./unfused_O1 && ./unfused_O1
./unfused_O2 && ./unfused_O2 && ./unfused_O2
./fused_O2 && ./fused_O2 && ./fused_O2

# 生成汇编
clang -O2 -S loop_fusion_test.c -o unfused_O2.s
clang -O2 -DFUSED=1 -S loop_fusion_test.c -o fused_O2.s
```

## 四、实验结果及分析和（或）源程序调试过程

### 4.1 实验 0：环境与基线

#### 4.1.1 运行结果

分别在 `-O0` 与 `-O2` 下编译运行 `toy_opt.c`，两个版本的终端输出均为：

```
101
```

【此处插入：`-O0` 与 `-O2` 运行输出截图，显示均输出 `101`】

多次运行 `time -p` 计时，由于程序本身逻辑极简，进程启动和 `printf` 开销占主导，`real` 时间中位数均在 **~0.02 s** 左右，两个版本差异不明显。

| 版本 | 编译选项 | 输出 | real 时间中位数 |
|------|----------|------|----------------|
| O0 | `-O0` | 101 | ~0.02 s |
| O2 | `-O2` | 101 | ~0.02 s |

#### 4.1.2 汇编对比分析

通过 `clang -O0 -S` 与 `clang -O2 -S` 生成的汇编文件差异显著，重点观察 `bar()` 函数：

**`-O0` 版 `bar()` 汇编**（完整保留局部变量、分支、内存读写）：

```asm
bar:
    sub    sp, sp, #16
    mov    w8, #100                        // =0x64
    str    w8, [sp, #8]                    // a = 10*10
    mov    w8, #20                         // =0x14
    str    w8, [sp, #4]                    // b = 20
    ldr    w8, [sp, #8]
    subs   w8, w8, #0
    cset   w8, le
    tbnz   w8, #0, .LBB0_2
    b      .LBB0_1
.LBB0_1:
    ldr    w8, [sp, #8]
    add    w8, w8, #1
    str    w8, [sp, #12]
    b      .LBB0_3
.LBB0_2:
    ldr    w8, [sp, #8]
    subs   w8, w8, #1
    str    w8, [sp, #12]
    b      .LBB0_3
.LBB0_3:
    ldr    w0, [sp, #12]
    add    sp, sp, #16
    ret
```

【此处插入：`toy_O0.s` 中 `bar:` 函数的截图】

**`-O2` 版 `bar()` 汇编**（常量折叠 + 死分支消除后仅剩一条 `mov`）：

```asm
bar:
    mov    w0, #101                        // =0x65
    ret
```

【此处插入：`toy_O2.s` 中 `bar:` 函数的截图】

**分析：**
- `-O2` 在编译期将 `10*10` 折叠为常量 `100`（常量折叠，Constant Folding）。
- 随后判断 `if (100 > 0)` 恒为真，`else` 分支成为死代码，被完全消除（Dead Branch Elimination）。
- 最终 `bar()` 直接返回常量 `101`。由于 `main()` 中 `x` 并非 `volatile` 的返回值使用，编译器甚至可以在 `main()` 中直接内联 `101` 到 `printf` 的参数，省略函数调用。

#### 4.1.3 思考题

**1. 为什么在 `-O2` 编译下 `10*10` 可能不会在运行时计算？**

答：因为编译器进行了**常量折叠（Constant Folding）**。`10` 和 `10` 均为编译期已知常量，其乘法结果 `100` 在编译阶段即可确定，无需生成运行时乘法指令。这是一种安全且无副作用的优化，因为纯常量的计算结果在任何执行环境下都相同。

**2. 为什么优化后程序输出仍然必须保持一致？**

答：C/C++ 编译器优化遵循 **"as-if" 规则**（as-if rule）。只要程序的可观察行为（observable behavior）——包括 I/O 操作、`volatile` 内存访问等——与未优化版本一致，编译器可以对内部计算过程做任何等价变换。本例中 `printf` 的输出仍是 `101`，用户无法从程序外部区分 `bar()` 是在运行时被调用还是在编译期被折叠，因此优化是合法的。

**3. 从汇编角度看，优化后代码通常有哪些变化？**

答：常见变化包括：
- **指令数显著减少**：如本例中 `bar()` 从十余条指令缩减为 2 条；
- **内存访问减少**：局部变量不再逐次 `str`/`ldr` 到栈，而是直接驻留寄存器；
- **分支扁平化**：死分支、恒真/恒假条件被消除，CFG（控制流图）更简洁；
- **函数内联**：小函数体被直接嵌入调用点，省去 `bl`/`ret` 开销。

#### 4.2.3 思考题

**1. 什么是 SIMD 向量化？它与普通循环执行有什么区别？**

答：**SIMD（Single Instruction, Multiple Data）向量化**是指使用宽向量寄存器（如 AArch64 NEON 的 128-bit 寄存器）在一次指令周期内对多组数据并行执行相同操作。普通标量循环每次迭代只处理一个数据元素（如一个 `float`），而 SIMD 循环一次可处理 4 个、8 个甚至更多元素。区别体现在：
- **执行宽度**：标量循环每迭代 1 个元素；SIMD 每迭代 N 个元素（N 取决于向量宽度 / 数据类型大小）；
- **内存访问模式**：SIMD 通常要求连续或对齐的内存加载，以使用 `ldr q` 等宽载入指令；
- **循环开销**：SIMD 循环的迭代次数减少为原来的 1/N，降低了分支和索引更新的开销。

**2. 自动向量化与 OpenMP 并行化分别属于哪一类并行？**

答：
- **自动向量化**属于**指令级 / 数据级并行（ILP / DLP）**。它在单个线程、单个核心内部通过 SIMD 功能单元并行处理数据，不增加线程数。
- **OpenMP 并行化**属于**线程级并行（TLP）**。它通过创建多个操作系统线程，将循环迭代分配到不同 CPU 核心上并行执行，利用多核资源。

**3. 在汇编代码中，哪些特征可以说明程序已经发生向量化？**

答：在 AArch64 汇编中，向量化通常表现为：
- **向量寄存器出现**：如 `v0.4s`、`v1.2d` 等带 `.` 后缀的寄存器命名；
- **宽载入/存储指令**：如 `ldr q0, [x0]`（加载 128-bit 到 `q0`）、`ldp q1, q4, [x11]`；
- **向量运算指令**：如 `fadd v0.4s, v0.4s, v2.4s`（向量浮点加）、`fmla v0.4s, v1.4s, v2.4s`（乘加）。

#### 4.3.3 汇编对比分析

**AoS（`transform_a_aos`）使用标量跨步访存：**

```asm
ldr     s1, [x11], #128          // 每次跨 128B 读取一个 float
ldr     s1, [x10], #128
...
```

【此处插入：`aos_soa_O3.s` 中 `transform_a_aos` 附近出现 `ldr s1, [x11], #128` 的截图】

由于 `S` 结构体大小为 128B（`1 float + 31 float pad`），访问 `x[i].a` 的步长即为 128B。编译器无法将离散的单点读取合并为连续向量加载，只能生成标量 `ldr s` 指令。

**SoA（`transform_a_soa`）使用向量连续访存：**

```asm
ldp     q1, q4, [x11, #-16]      // 连续载入 128-bit 向量寄存器
...
ldr     q5, [x10], #32
ldr     q4, [x9], #32
```

【此处插入：`aos_soa_O3.s` 中 `transform_a_soa` 附近出现 `ldp q` 或 `ldr q` 的截图】

SoA 将 `a` 字段抽出为独立的连续 `float[]` 数组，访存模式完全连续且对齐，编译器成功触发向量化，使用 `ldp q`（一对 128-bit 向量加载）批量读取数据。

**分析：**
- AoS 的跨步访存导致每加载一个有效数据就要跳过 31 个填充字节的 Cache line，Cache 利用率极低，且无法触发 SIMD 向量化；
- SoA 的连续访存充分利用了空间局部性，Cache line 中几乎全是有效数据，同时满足 NEON 向量加载的对齐和连续性要求，因此性能显著优于 AoS。

#### 4.3.4 思考题

**1. 为什么 AoS（Array of Structs）访问可能不利于向量化？**

答：AoS 将多个字段打包在同一个结构体中连续存放。当程序只访问其中一个字段（如本例的 `a`）时，相邻迭代访问的内存地址之间的步长等于整个结构体的大小（本例为 128B）。这种大跨步访存破坏了 SIMD 所需的**连续内存假设**：向量加载指令（如 `ldr q`）期望一次读取连续地址的多个元素，而 AoS 的有效数据被填充字段隔开，编译器无法安全地将其合并为宽向量操作，只能退化为标量 `ldr`。

**2. SoA（Struct of Arrays）为什么更容易形成连续访存？**

答：SoA 将同一字段的所有值单独提取为一个连续数组。例如，原本 `S xs[N]` 中的 `xs[i].a` 被重组为 `float a[N]`。此时访问 `a[i]` 的步长仅为 `sizeof(float) = 4B`，内存完全连续。连续数组满足 SIMD 加载指令的地址连续性要求，编译器可以轻松地向量化，并最大化 Cache line 的有效利用率。

**3. 在汇编中，连续访存通常会表现出哪些指令特征？**

答：连续访存在 AArch64 汇编中的典型特征包括：
- **宽向量加载/存储**：如 `ldr q0, [x0]`（128-bit 加载）、`ldp q1, q2, [x0]`（加载两个 128-bit 向量）；
- **后变址寻址**：如 `ldr q5, [x10], #32`，每次加载后自动将地址递增 32B，适合循环内连续读取；
- **向量运算**：如 `fadd v0.4s, v1.4s, v2.4s`，表明加载上来的数据被直接用于 SIMD 计算。

#### 4.4.3 思考题

**1. 为什么输入数据分布会影响 PGO 优化效果？**

答：PGO 的核心机制是“以实际运行 profile 指导编译决策”。编译器根据训练运行中各分支的**真实 taken / not-taken 比例**来重排代码：高频路径被尽量放在连续地址，以减少跳转和指令 Cache miss。如果训练数据与真实运行数据的分布不同（例如训练时分支 A 常走，真实运行时分支 B 常走），编译器会错误地将低频路径优化为 fall-through，导致真实运行时反而增加跳转开销，产生“负优化”。

**2. PGO 为什么可以改善分支预测或代码布局？**

答：
- **代码布局**：编译器将高频执行的基本块放在相邻地址，使得 CPU 取指单元可以顺序预取，减少指令 TLB miss 和 I-Cache miss。
- **分支预测**：虽然现代 CPU 有动态分支预测器，但编译期静态布局仍可与硬件预测协同：将最可能的分支方向设为 fall-through（不跳转），即使硬件预测失败，不跳转的代价通常也低于跳转。

**3. 如果训练数据与真实运行数据差异很大，PGO 可能产生什么问题？**

答：会产生**负优化（Pessimization）**。具体表现为：
- 代码布局偏向训练数据中的高频路径，而真实运行中这些路径反而低频；
- 函数内联/外联决策基于错误的调用频率，导致代码膨胀或 Cache 污染；
- 总体性能可能低于未使用 PGO 的基线版本。因此，PGO 的有效性高度依赖“训练数据具有代表性”这一前提。

#### 4.5.3 思考题

**1. Autotuner 的基本思想是什么？**

答：Autotuner 的基本思想是**将编译优化参数空间视为搜索空间，以实际运行反馈为评价指标，自动寻找较优的编译配置**。具体流程为：
- **Generate**：编译器在 `-fautotune-generate` 下暴露出一组可调的优化参数（如内联阈值、循环展开因子、向量化宽度等）；
- **Iterate**：在每一轮中，Autotuner 根据当前搜索策略选择一组参数，编译程序并运行，将运行时间（或其他指标）作为反馈；
- **Finalize**：经过多轮迭代后，将历史最优参数固化到 `config.yaml`，后续编译直接使用该配置。

**2. 为什么自动调优有时只带来很小的性能提升？**

答：原因包括：
- **基线已较优**：现代编译器的 `-O2`/`-O3` 已经包含了大量经过验证的通用优化，剩余可调整参数的收益空间有限；
- **搜索空间庞大但局部最优多**：编译参数之间存在复杂的交互关系，Autotuner 的启发式搜索可能陷入局部最优，难以找到全局最佳；
- **运行环境噪声**： benchmark 的运行时间受系统负载、Cache 状态、CPU 频率调节等噪声影响，少量性能差异可能无法被稳定区分。

**3. 为什么在不同机器上 Autotuner 的结果可能不同？**

答：编译优化效果高度依赖目标微架构特性，包括：
- **Cache 层次与容量**：不同的 L1/L2/L3 Cache 大小决定了循环展开和分块（tiling）的最佳参数；
- **分支预测器结构**：不同的预测器对代码布局的敏感度不同；
- **SIMD 宽度**：ARMv8 为 128-bit NEON，ARMv9 可能支持 128/256-bit SVE，向量化策略完全不同；
- **内存带宽与延迟**：影响内存密集型的 prefetch 和排布决策。
因此，同一组参数在服务器级 ARM 芯片与嵌入式芯片上的表现可能截然相反，Autotuner 必须针对具体硬件重新搜索。

#### 4.6.3 思考题

**1. 什么是 Loop Fusion？为什么循环合并可能减少内存访问开销？**

答：**Loop Fusion（循环融合）**是指将两个或多个相邻的、具有相同迭代域的循环合并为一个循环。合并后，原来需要多次遍历的数组现在只需遍历一次。减少内存访问开销的原因在于：
- **Cache 复用**：`a[i]` 在一次循环中被读取后，可以直接用于两个计算（`c[i]` 和 `d[i]`），无需在第二个循环中重新从内存或 L2 Cache 加载；
- **写回合并**：`c[i]` 和 `d[i]` 的写操作如果落在同一 Cache line，可以更紧凑地利用 Write Buffer；
- **循环开销降低**：减少了一次完整的循环控制（初始化、条件判断、索引递增、分支回跳）开销。

**2. 什么是 Strength Reduction？为什么用地址递推可以替代乘法索引？**

答：**Strength Reduction（强度削弱）**是一种将高开销运算替换为低开销等价运算的优化。在循环中，最常见的应用是将数组索引的乘法 `base + i * sizeof(T)` 替换为地址递推：初始化指针为 `base`，每次迭代执行 `ptr += sizeof(T)`。

地址递推替代乘法的优势：
- **指令延迟**：在大多数 CPU 上，`add` 指令的延迟（1 cycle）远低于 `mul`（3~5 cycles）；
- **地址生成单元**：现代 CPU 有专门的地址生成AGU，可以在 load/store 流水线阶段并行完成 `ptr + offset`，不占用 ALU；
- **循环 carried dependency 更轻**：`ptr += step` 的依赖链比 `i * step` 更短，有利于指令级并行。

**3. 为什么这些循环优化通常在较高优化等级（如 `-O2`）才会出现？**

答：循环融合、强度削弱等优化需要编译器进行复杂的程序分析，包括：
- **别名分析（Alias Analysis）**：确认两个循环访问的数组无重叠，融合后不会引入新的数据依赖或破坏原有语义；
- **数据依赖分析**：判断循环体内语句之间的读写依赖关系，决定是否允许重排或合并；
- **控制流图变换**：合并循环需要重构 CFG，可能引入新的基本块和入口/出口边；
- **寄存器压力估算**：融合后循环体变大，局部变量增多，编译器需要评估寄存器分配是否仍可行。

这些分析计算量大，且 `-O0`/`-O1` 的设计目标更偏向快速编译和保留调试信息，因此通常仅在 `-O2` 及以上才会启用。

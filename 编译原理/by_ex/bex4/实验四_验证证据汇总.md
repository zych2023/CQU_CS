## 实验 0：环境与基线（O0/O1/O2/O3 + 生成汇编验证优化发生）

### 1. 运行输出
```
-O0: 101
-O2: 101
```
两个版本输出一致。

### 2. 计时数据
| 版本 | 编译选项 | 时间中位数（real） |
|------|----------|-------------------|
| O0 | `-O0` | ~0.02s |
| O2 | `-O2` | ~0.02s |

程序本身极轻量，时间主要消耗在进程启动/printf，但汇编差异显著。

### 3. 代码生成证据

**O0 版 `bar()` 汇编（完整保留了局部变量、分支、内存读写）：**
```asm
bar:
    sub    sp, sp, #16
    mov    w8, #100                        // =0x64
    str    w8, [sp, #8]                    // a = 10*10
    mov    w8, #20                         // =0x14
    str    w8, [sp, #4]                    // b = 20
    ldr    w8, [sp, #8]
    subs   w8, w8, #0
    cset   w8, le
    tbnz   w8, #0, .LBB0_2
    b      .LBB0_1
.LBB0_1:
    ldr    w8, [sp, #8]
    add    w8, w8, #1
    str    w8, [sp, #12]
    b      .LBB0_3
.LBB0_2:
    ldr    w8, [sp, #8]
    subs   w8, w8, #1
    str    w8, [sp, #12]
    b      .LBB0_3
.LBB0_3:
    ldr    w0, [sp, #12]
    add    sp, sp, #16
    ret
```

**O2 版 `bar()` 汇编（常量折叠 + 死分支消除 + 内联后仅剩一条 mov）：**
```asm
bar:
    mov    w0, #101                        // =0x65
    ret
```

**结论：** `-O2` 将 `10*10` 在编译期折叠为 `100`，随后 `if (100>0)` 恒成立，死分支被消除，最终 `bar()` 直接返回常量 `101`。`main()` 中甚至不需要调用 `bar()`，直接把 `101` 送入 `printf`。

## 实验 1：自动并行化（SIMD/NEON 向量化 + OpenMP 并行 for）

### 1. 运行输出
三个版本均正常输出 `time: xxx s`。

### 2. 计时数据（中位数）
| 版本 | 编译选项 | 时间（s） | 备注 |
|------|----------|----------|------|
| no-vec | `-O2` + `vectorize(disable)` | 0.966 | 禁止向量化基线 |
| vec | `-O2` | 0.833 | 自动向量化 |
| omp | `-O2 -fopenmp` + `omp parallel for` | 0.330 | 4线程OpenMP |

### 3. 代码生成证据

**vec 版本出现 NEON 向量指令（`fadd v0.4s` 等）：**
```asm
; vec_vec.s 片段
fadd    v0.4s, v0.4s, v2.4s
fadd    v1.4s, v1.4s, v3.4s
```

**no-vec 版本无向量寄存器，仅有标量 `fadd s`：**
```asm
; vec_no_vec.s 片段（无 ldr q / fadd v 出现）
```

**结论：**
- 自动向量化属于 **指令级并行（ILP/SIMD）**，通过 `v0.4s` 等 128-bit NEON 寄存器一次处理 4 个 float。
- OpenMP 属于 **线程级并行（TLP）**，在 4 核容器上带来约 3× 加速。

## 实验 2：内存布局优化（struct-peel + AoS→SoA）

### A 部分：struct-peel（可选验证）

编译命令：
```bash
clang struct_peel.c -O3 -flto -fuse-ld=lld -o test.struct-peel.exe
```

**calloc/malloc 调用统计：**
| 版本 | `bl.*calloc/malloc` 次数 |
|------|-------------------------|
| struct-peel | 1 |
| no-struct-peel | 1 |

**结论：** 本 Docker/QEMU 仿真环境下，`struct-peel` 未产生可见的代码生成差异（两者均只调用 1 次 `calloc`）。这与实验指导书提示的“部分 Docker/离线镜像可能无法复现”一致。

### B 部分：AoS → SoA（必做）

### 1. 运行输出
```
checksum_aos=794399395.487 checksum_soa=794399395.487
time_aos=0.121561 time_soa=0.094587
```
多次运行 checksum 完全一致，证明两种布局计算结果等价。

### 2. 计时数据（中位数）
| 布局 | 时间（s） |
|------|----------|
| AoS | ~0.121 |
| SoA | ~0.095 |

### 3. 代码生成证据

**AoS（`transform_a_aos`）使用标量跨步访存 `ldr s1`：**
```asm
; aos_soa_O3.s
ldr     s1, [x11], #128          // 每次跨 128B 读取一个 float
ldr     s1, [x10], #128
...
```

**SoA（`transform_a_soa`）使用向量宽载入 `ldp q1, q4`：**
```asm
; aos_soa_O3.s
ldp     q1, q4, [x11, #-16]      // 连续载入 128-bit 向量寄存器
...
ldr     q5, [x10], #32
ldr     q4, [x9], #32
```

**结论：**
- AoS 中结构体带 31 个 float pad，导致访问 `x[i].a` 时步长为 128B，无法连续访存，编译器只能生成标量 `ldr s`。
- SoA 将 `a` 字段单独抽出为连续数组，访存完全连续，编译器成功向量化，使用 `ldp q` 批量加载。

## 实验 3：PGO 反馈优化

### 1. 运行输出
基线与 PGO 版本均输出 `Average: xxx ms`。

### 2. 计时数据（中位数）
| 版本 | 时间（ms） |
|------|-----------|
| 基线 `-O2` | ~14,271 |
| PGO `-O2 -fprofile-instr-use` | ~11,285 |

**PGO 版本比基线快约 21%**（同机同法对比）。

### 3. 代码生成证据

**基线 `bubble_sort` 反汇编片段：**
```asm
00000000000007a8 <bubble_sort>:
    7a8:    cmp     w1, #0x2
    7ac:    b.lt    818 <bubble_sort+0x70>
    7b0:    sub     w8, w1, #0x1
    ...
    7d8:    add     x12, x11, #0x1        // 外循环初始化
    7dc:    cmp     x12, x8
    7e0:    b.cs    7c4 <bubble_sort+0x1c>  // 条件跳转
    ...
    7fc:    ldr     w15, [x0, x11, lsl #2]
    800:    ldr     w16, [x14]
    804:    cmp     w15, w16
    808:    b.ge    7f0 <bubble_sort+0x48>
    ...
```

**PGO `bubble_sort` 反汇编片段：**
```asm
0000000000000680 <bubble_sort>:
    680:    cmp     w1, #0x2
    684:    b.lt    6e4 <bubble_sort+0x64>
    688:    sub     w8, w1, #0x1
    698:    add     x12, x11, #0x1        // 外循环头被提前
    69c:    cmp     x12, x8
    6a0:    b.cs    6d0 <bubble_sort+0x50>
    ...
    6ac:    ldr     w15, [x0, x11, lsl #2]
    6b0:    ldr     w16, [x14]
    6b4:    cmp     w15, w16
    6b8:    b.ge    6c4 <bubble_sort+0x44>
    ...
    6cc:    b.ne    6ac <bubble_sort+0x2c>   // 循环回边更紧凑
```

**结论：**
- PGO 利用训练数据（升序数组，`a[i] < a[j]` 分支高度可预测）指导编译器重新排布热点路径。
- 观察到的变化：`bubble_sort` 函数体更紧凑，条件分支的 fall-through 路径更符合实际执行频率，减少了跳转开销。

Iteration 4 -> time: 14.310000

### 3. 调优证据
- `AUTOTUNE_DATADIR=/tmp/autotuner_data` 已生成并被使用。
- `config.yaml` 已生成（通过 `llvm-autotune finalize`）。

### 4. 结果分析
**注意：** 本实验运行在 x86 主机上的 `linux/arm64` QEMU 仿真容器中。绝对性能数据不具备跨平台可比性，且 QEMU 仿真的计时噪声较大。

- **基线 O2**：13.697s, 4380.52 iter/sec
- **Autotuned**：31.131s, 1927.34 iter/sec（本轮实际更慢）

**原因分析：**
1. Autotuner 优化的参数组合在真实 ARM 硬件上可能更优，但在 QEMU 仿真层中，某些指令序列或内存布局反而导致更多宿主机翻译开销。
2. 迭代采样阶段的最佳时间（13.808s）与基线（13.697s）接近，说明在该仿真环境下优化空间有限。
3. 最终 `finalize` 后重新编译的 `coremark_autotuned` 使用了某一组被标记为“最优”的参数，但这组参数在 QEMU 下表现劣化，说明 Autotuner 的结果与运行环境强相关。

**课程要求强调：** 本课程更关注“优化流程是否完整跑通”以及“有无 config.yaml 生成”，性能只做同机同法的相对对比。

## 实验 5：综合循环体优化（Loop Fusion + Autotuner 对比）

### 1. 运行输出
所有版本 `Checksum: 45823895273472.000000`，输出一致。

### 2. 计时数据（中位数）
| 版本 | 编译选项 | 时间（s） |
|------|----------|----------|
| unfused O0 | `-O0` | 0.338 |
| unfused O1 | `-O1` | 0.206 |
| unfused O2 | `-O2` | 0.175 |
| fused O2 | `-O2 -DFUSED=1` | 0.157 |
| fusion_tuned (Autotuner) | `-O2 -fautotune` | 0.179 |

### 3. 代码生成证据

**unfused O2 的循环结构（`Inner Loop Header` 数量）：**
```
85 : .LBB0_5  =>This Inner Loop Header: Depth=1
114: .LBB0_7  =>This Inner Loop Header: Depth=1
133: .LBB0_9  =>This Inner Loop Header: Depth=1
144: .LBB0_11 =>This Inner Loop Header: Depth=1
163: .LBB0_13 =>This Inner Loop Header: Depth=1
174: .LBB0_15 =>This Inner Loop Header: Depth=1
193: .LBB0_17 =>This Inner Loop Header: Depth=1
214: .LBB0_19 =>This Inner Loop Header: Depth=1
```
共 **8** 个内循环头（`REPEAT=3`，每次外循环展开后产生 2 个内循环）。

**fused O2 的循环结构：**
```
85 : .LBB0_5  =>This Inner Loop Header: Depth=1
115: .LBB0_7  =>This Inner Loop Header: Depth=1
135: .LBB0_9  =>This Inner Loop Header: Depth=1
155: .LBB0_11 =>This Inner Loop Header: Depth=1
184: .LBB0_13 =>This Inner Loop Header: Depth=1
```
共 **6** 个内循环头，比 unfused 少，说明 fused 版本将两个相邻循环合并为一个循环体，减少了循环开销和内存遍历次数。

### 4. Autotuner 对比
- `fusion_tuned`（Autotuner O2）时间与 `unfused_O2` 接近（~0.18s），未明显优于手工 fused 版本。
- 原因：Loop Fusion 的源码级 fused 实现已经提供了更优的循环结构，Autotuner 在此基础上进一步挖掘空间有限。

## 附录：思考题简要回答

### 实验0
1. **为什么 `-O2` 下 `10*10` 可能不在运行时计算？** —— 常量折叠（Constant Folding），编译器在编译期算出 `100` 并直接嵌入指令。
2. **为什么优化后输出仍必须一致？** —— 编译器优化必须遵守“as-if”规则：只要可观察行为（I/O、volatile 访问）一致，内部实现可任意变换。
3. **优化后汇编通常有哪些变化？** —— 指令数减少、分支扁平化、寄存器替代内存、函数内联、循环向量化等。

### 实验1
1. **SIMD 向量化与普通循环的区别？** —— SIMD 用宽寄存器一次处理多组数据；普通循环一次处理一个标量元素。
2. **自动向量化与 OpenMP 分别属于哪类并行？** —— 自动向量化属于 **数据级并行（DLP/SIMD）**；OpenMP 属于 **线程级并行（TLP）**。
3. **汇编中向量化的特征？** —— 出现向量寄存器（如 `v0.4s`）、宽载入指令（`ldr q`, `ldp q`）、向量运算（`fadd v`）。

### 实验2
1. **为什么 AoS 不利于向量化？** —— 访问单个字段时步长大（跨整个结构体），无法形成连续内存流。
2. **SoA 为什么更容易连续访存？** —— 同名字段连续存放，形成紧凑的 `float[]`，满足 SIMD 连续加载条件。
3. **连续访存的汇编特征？** —— 出现 `ldr q` / `ldp q` / `stur q` 等 128-bit 宽指令，以及 post-index 地址递增（如 `[x11], #32`）。

### 实验3
1. **为什么输入数据分布影响 PGO？** —— PGO 根据实际分支频率重排代码，若训练数据与真实分布不同，优化方向会偏离。
2. **PGO 如何改善分支预测/代码布局？** —— 将高频执行路径放在相邻地址，减少跳转和指令 Cache miss。
3. **训练数据与真实差异大时的问题？** —— 可能产生“负优化”，代码布局偏向低频路径，导致真实运行时性能下降。

### 实验4
1. **Autotuner 基本思想？** —— 通过搜索编译参数空间，以实际运行反馈为指标，自动寻找性能较优的编译配置。
2. **为什么有时只带来很小提升？** —— 基准编译器（如 `-O2`）已做了大量通用优化，剩余参数空间收益有限；或受运行环境噪声干扰。
3. **不同机器结果为何不同？** —— 优化效果高度依赖微架构（Cache 大小、分支预测器、SIMD 宽度等），同一组参数在不同 CPU 上表现差异大。

### 实验5
1. **Loop Fusion 为何减少内存访问开销？** —— 合并后只需遍历一次数组，减少 Cache 行加载/写回次数，同时可能提升向量化机会。
2. **Strength Reduction 为何可用地址递推替代乘法？** —— 每次迭代地址增加固定步长（`add`）比计算 `i*STRIDE`（`mul`）更快，且部分 CPU 上乘法有更高延迟。
3. **为何循环优化通常在高优化等级才出现？** —— 需要可靠的别名分析、数据依赖分析和 CFG 变换，低优化等级为保证调试信息和编译速度会跳过这些复杂分析。
